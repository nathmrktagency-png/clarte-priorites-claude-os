\# MODE D’EMPLOI — CLAUDE OS CLARTÉ & PRIORITÉS MARKETING v1

\#\# 1\. Rôle du système

Ce système Claude sert à aider Clarté & Priorités Marketing à :  
\- structurer le business ;  
\- prospecter sobrement ;  
\- qualifier les prospects ;  
\- vendre l’audit payant ;  
\- produire des audits ;  
\- prioriser les actions ;  
\- préparer des dispositifs cadrés ;  
\- créer des livrables ;  
\- suivre les KPI ;  
\- protéger le scope ;  
\- respecter la confidentialité ;  
\- éviter toute promesse de ROI.

Il ne sert pas à :  
\- garantir des ventes ;  
\- promettre un ROI ;  
\- agir comme agence 360 ;  
\- envoyer automatiquement des messages ;  
\- livrer automatiquement des documents ;  
\- facturer automatiquement ;  
\- remplacer une validation juridique, comptable ou humaine.

\#\# 2\. Règle source de vérité

Le fichier prioritaire est :

00\_MASTER\_CONTEXT\_CLAUDE

Si un autre document contredit le Master Context :  
1\. appliquer le Master Context ;  
2\. signaler la contradiction ;  
3\. proposer une correction ;  
4\. ne rien modifier sans validation humaine.

\#\# 3\. Règle générale de confidentialité

Un client réel \= un Project Claude séparé \+ un dossier Drive séparé.

Ne jamais mélanger plusieurs clients.  
Ne jamais mettre de données client réelles dans les Projects internes.  
Ne jamais mettre de données client dans GitHub.  
Ne jamais envoyer, publier, livrer ou facturer sans validation humaine.

\#\# 6\. Workflow prospect

1\. Identifier un prospect.  
2\. L’analyser dans 01\_PROSPECTION.  
3\. Vérifier la compatibilité.  
4\. Préparer un message sobre.  
5\. Ne pas donner de diagnostic gratuit.  
6\. Proposer une conversation courte.  
7\. Qualifier.  
8\. Proposer l’audit payant si compatible.  
9\. Refuser ou requalifier si non compatible.

\#\# 7\. Workflow client

1\. Prospect compatible.  
2\. Audit vendu.  
3\. Paiement initial reçu.  
4\. Création du Project client.  
5\. Création du dossier Drive client.  
6\. Collecte des informations.  
7\. Audit.  
8\. Matrice de priorisation.  
9\. Rapport.  
10\. Restitution.  
11\. PV de décision.  
12\. Gate audit → dispositif.  
13\. Dispositif seulement si justifié.  
14\. Livraison après contrôle qualité et solde.  
15\. Accompagnement éventuel.  
16\. Archivage.

\#\# 8\. Gate audit → dispositif

Un dispositif est interdit si :  
\- audit non vendu ;  
\- paiement initial non reçu ;  
\- données non collectées ;  
\- rapport non terminé ;  
\- restitution non faite ;  
\- PV non validé ;  
\- priorité non issue de l’audit ;  
\- client demande une exécution directe ;  
\- demande hors-scope non cadrée.

Phrase à utiliser :  
GATE NON VALIDÉ — dispositif non autorisé à ce stade.

\#\#\# Vérification livrable

Vérifie ce livrable avant envoi client.  
Contrôle :  
\- Master Context ;  
\- absence de promesse ROI ;  
\- confidentialité ;  
\- scope ;  
\- ton de marque ;  
\- informations inventées ;  
\- éléments à vérifier ;  
\- validation humaine.

\#\#\# Gate dispositif

Vérifie si le gate audit → dispositif est validé.  
Réponds avec :  
1\. conditions validées ;  
2\. conditions manquantes ;  
3\. risques ;  
4\. décision ;  
5\. prochaine étape.

\#\# 12\. Règle anti-dispersion

Avant de créer un nouvel outil, un nouveau Project, une nouvelle Skill ou une nouvelle automatisation, vérifier :

1\. Est-ce vraiment utile maintenant ?  
2\. Est-ce que le workflow manuel est stable ?  
3\. Est-ce que cela réduit une friction réelle ?  
4\. Est-ce que cela protège le temps et la marge ?  
5\. Est-ce que cela augmente un risque de confidentialité ?  
6\. Est-ce que cela peut être testé sans données sensibles ?

Si la réponse est floue : ne pas automatiser.
